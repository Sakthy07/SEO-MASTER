TAB 1 / PRE-HOST

Edits inside the codebase

All changes happen in .tsx source files; delivered to the developer as a diff/branch; built once into the static export and uploaded to Hostinger.

01ON-PAGE

On-Page SEO

Per-route content hygiene — what each page says about itself in the head and how the body is structured.

WHERE app/[route]/page.tsxHOW metadata + JSX
  • Title tags — unique 50–60 char title per page in the metadata.title field of each page.tsx. Next.js compiles it to <title>.
  • Meta descriptions — 140–160 char metadata.description per page. Compiled to <meta name="description">.
  • One H1 per page + clean H2/H3 hierarchy in the JSX. No skipped levels.
  • Primary keyword placed once in title, H1, first 100 words, slug, OG title. No stuffing (<2% density).
  • Image alt + filenames — descriptive alt on every <Image> + renamed source files.
  • Clean URL slugs — folder name under app/ = the URL.
  • Internal linking — 3–5 descriptive <Link> elements per page using next/link.
02TECHNICAL

Technical SEO

The site's plumbing — config and build-level decisions that shape crawlability and Core Web Vitals.

WHERE next.config.mjs · app/ · componentsHOW config + components
  • Static export config — output: 'export' + images: { unoptimized: true } in next.config.mjs so the build produces the out/ folder Hostinger can serve.
  • Image dimensions — explicit width/height on every <Image> to prevent CLS.
  • JS defer/async — third-party scripts loaded via next/script with strategy="afterInteractive" or lazyOnload.
  • sitemap.ts — new file at app/sitemap.ts; Next generates /sitemap.xml at build.
  • robots.ts — new file at app/robots.ts; Next generates /robots.txt at build.
  • Canonical tags — metadata.alternates.canonical per route, absolute URL.
  • Mobile-friendly — Tailwind responsive utilities; verified via Lighthouse later.
03HTML PROTOCOLS

Advanced HTML Protocols

Everything that lives in <head> beyond the basics — controls previews and engine signals.

WHERE app/layout.tsx + each page.tsxHOW Metadata API
  • Open Graph — metadata.openGraph with title, description, url, type, locale=en_IN, siteName, 1200×630 image.
  • Twitter Cards — metadata.twitter with card: "summary_large_image", site handle, title, description, image.
  • Resource hints — next/font auto preconnect + priority flag on LCP <Image> auto preload.
  • Lang — change <html lang="en"> to lang="en-IN" in app/layout.tsx.
  • Robots meta — metadata.robots = { index: true, follow: true } globally.
  • Theme-color + referrer — set in layout metadata.
04SCHEMA · JSON-LD

Main + Sub Schemas

Structured data injected as <script type="application/ld+json"> into each page's server-rendered HTML.

WHERE app/layout.tsx + per-page + components/seoHOW JSON-LD components
  • Main — Organization on homepage: legal name, logo, URL, contactPoint, sameAs, address.
  • Main — WebSite on homepage: url, name, potentialAction.
  • Sub — LocalBusiness on /contact: full NAP, geo, openingHours, telephone.
  • Sub — Service on /services + /solutions: serviceType, areaServed, provider, offers.
  • Sub — FAQPage wherever FAQs exist (3–5 Q&As each).
  • Sub — BreadcrumbList on every inner route.
05BREADCRUMBS

Breadcrumbs

Both the visible nav trail and the matching schema so Google can show the crumb in search.

WHERE components/ui/breadcrumb.tsxHOW shared component
  • Visible breadcrumb nav — new <Breadcrumb> component with aria-label="breadcrumb", driven by the route segments.
  • BreadcrumbList schema — same component emits matching JSON-LD on every page it renders on.
  • Mounted in app/layout.tsx just below the navbar, hidden on the homepage.
06AIEO + GEO · CODE PART

AI-Engine & Entity Foundation

The code-side prep that lets AI engines recognise the brand. The live-side activation happens in Tab 2.

WHERE lib/site.ts + layout.tsx + page bodiesHOW schema + content
  • sameAs array — real social + GBP URLs wired into Organization schema.
  • Entity facts — founding year, legal name, services, area, in lib/site.ts.
  • GEO content format — 1-sentence service definitions + brand stats added in JSX so AI engines have quotable hooks.
  • en-IN locale on both <html> and OG locale.
Wikidata submission, Reddit/Quora seeding, and brand-mention monitoring all happen after hosting — see Tab 2.

TAB 3 / END-TO-END

How the loops actually run

Three engines that keep growing the brand after launch. Each shown step by step — what we do, where it happens, and how it loops back.

FLOW 01

AEO · AIEO · GEO Loop

monthly cycle · 4 AI engines
We ask the four AI engines real customer questions to see where the brand stands today, apply what we learn back into the website, seed our presence off-site where AI engines crawl, then re-test next month to confirm growth.
1
Ask the AI engines
Week 2 baseline · monthly
  • 20 customer questions × 4 engines = 80 tests across ChatGPT, Perplexity, Google AI Overviews, Gemini.
  • Driven via Claude in Chrome; screenshots saved to 05_aeo_geo/.
  • Capture citation rate (AEO), mention rate (AIEO), entity recognition rate (GEO) per engine.
  • Log competitors cited + their content format (stat / definition / list / FAQ).
  • Output → 03_citation-tracker.xlsx, 09_geo-tracker.xlsx, 10_entity-tracker.xlsx.
2
Apply on the website
code edits → rebuild → redeploy
  • Add FAQ sections + FAQPage schema for the questions AI engines answered well (and badly).
  • Write AI-friendly blog posts — definitions, stat blocks, lists, Q&A formatted.
  • Insert citation hooks in body copy (short, quotable, dateable facts).
  • Keep Organization + LocalBusiness + sameAs schema current.
  • Run Task 6 — GEO Content Format Check on existing pages to reformat them for citation.
3
Seed off-site
Week 2 setup · continuous
  • Post helpful seed answers on Reddit + Quora in our niche — AI engines crawl these heavily.
  • Submit Wikidata brand entry (after seasoning the account 3–7 days).
  • Publish a Perplexity Page using one of our cornerstone articles.
  • Activate brand-mention monitor (Google Alerts / Mention.com) → alerts feed 10_geo_aieo/brand-mentions-log.xlsx.
  • Run Task 7 — Entity Audit: Knowledge Panel, Wikidata QID, sameAs propagation, directory listings.
4
Re-test & measure
monthly 1st · monthly 15th
  • Monthly 1st — re-run the same 80-row test → diff against last month's scores.
  • Monthly 15th — brand-mention + Reddit/Quora review; new mentions logged + outreach drafted.
  • Citation / mention / recognition rates should climb every cycle.
  • Pages losing citations get queued back to Step 2 for a refresh.
  • Trend visible in the weekly Monday report + Looker dashboard.
Loop back to Step 2

Every month, the new test reveals which pages slipped, which competitors gained, and which engines need work. Those become next month's code edits — and the cycle continues forever.

↺
✓
End-to-end outcome: citation rate climbs (e.g. 8 → 28 / 80), brand-mention rate climbs (5% → 35%), entity-recognition climbs (2/10 → 8/10) over the first 90 days. The four AI engines start treating the brand as a known, quotable source.
FLOW 02

Review Automation

daily 10 AM · WhatsApp + GBP
Once the GBP is live and jobs start completing, the daily 10 AM engine takes over: it asks happy customers for reviews, sweeps the live GBP for new ones, drafts replies for owner approval, and feeds everything back into the weekly report.
1
Setup (one-time, Week 3)
Task 10
  • Customise 3 templates in client voice — review-request-1.txt, review-request-2.txt, review-thanks.txt — in 07_outreach/templates/.
  • Choose send channel — WhatsApp Web (free, ≤15/day) or Wati/Interakt API (paid, no cap).
  • Generate the Google review link (g.page/r/…) and paste it into 00_context/about.md.
  • Seed 5 recent customers into the "Pipeline" sheet of 06_review-tracker.xlsx + dry-run.
  • Add Daily 10 AM calendar reminder — triggers the daily run.
2
Daily 10 AM run
every working day
  • Open 06_review-tracker.xlsx → "Pipeline" sheet.
  • Job complete + 1 day → draft Request 1 (substitute name/service/technician/link).
  • Request 1 sent + 5 days, no review → draft Request 2.
  • Review posted → draft Thanks message.
  • Request 2 + 14 days no reply → mark "No response", stop.
  • Owner approves the first send of each batch; thereafter messages send within the WhatsApp cap.
3
New-review sweep
same daily run
  • Query Semrush Local for new GBP reviews in the last 24 hours.
  • Append to "Reviews" sheet — date, platform, customer, stars, text, auto-classified sentiment.
  • Match the reviewer back to the Pipeline row → fill Review Posted Date + Stars.
  • For each new review → draft a personalised owner-reply (50–100 words). For negatives → acknowledge + apologise + offer offline resolution.
  • Owner approves + posts the reply via GBP.
4
Log & report
daily + weekly
  • Each daily run logs counts to 08_reports/changelog-YYYY-MM-DD.md — Requests 1/2 sent, Thanks sent, new reviews captured, errors.
  • Monday 9 AM weekly report summarises the week's review activity for the client.
  • Looker Studio dashboard shows the review count trend + average rating.
Loop back to Step 2

The pipeline never empties — new completed jobs flow in continuously from the client's job sheet, keeping the daily engine running. Negative-review handling escalates to live owner conversation.

↺
✓
End-to-end outcome: review count grows from the baseline (e.g. 14 → 46 in 90 days) without owner chasing customers; average star rating stays high because asks go only to happy "job complete" customers; replies are timely and on-brand.
FLOW 03

Tracking — End to End

GA4 + GTM + GSC + Looker
Set up once (Week 1, Day 2), then runs forever. Every signal — visitors, conversions, search clicks, AI traffic, brand mentions — flows into one Looker dashboard the client opens anytime.
1
Setup (one-time, Week 1 Day 2)
Task 9
  • Create GA4 property under the digi_design account → record Measurement ID in 00_context/tracking-checklist.md.
  • Install GTM container on the live site → verify it fires in GA4 DebugView.
  • Define conversions: form submit, phone click, WhatsApp click, GBP click.
  • Verify Search Console via DNS; submit sitemap (/sitemap.xml).
  • Create AI Referral segment in GA4 → source matches chatgpt|perplexity|gemini|claude|bing-chat|copilot.
  • Activate brand-mention monitor (Google Alerts / Mention.com).
  • Clone Looker Studio template → swap data sources to this client's GA4 + GSC.
2
Data flows in (live, continuous)
from launch onward
  • Visitor lands → GTM fires → GA4 records session, source, device, geography.
  • Visitor converts → the matching event (form / phone / WhatsApp / GBP-click) increments.
  • Google crawls + indexes → GSC populates clicks, impressions, position, Core Web Vitals.
  • Visitor arrives from ChatGPT / Perplexity / Gemini → tagged automatically into the AI Referral segment.
  • Brand mentioned online → monitor email arrives + logged in 10_geo_aieo/brand-mentions-log.xlsx.
3
Aggregate in Looker
live, refreshes hourly
  • Looker dashboard pulls GA4 + GSC + AI Referral segment + brand-mention velocity into one view.
  • Charts: organic sessions trend · top pages · conversions by source · AI Referral channel · GSC clicks & positions · review count.
  • Dashboard URL saved in 00_context/about.md → client opens it anytime, no login friction.
  • Shared with the client at Day-30 handoff (Week 4).
4
Weekly & monthly synthesis
Mon 9 AM · 1st of month
  • Mon 9 AM Weekly Report auto-generated → emails the client a written summary (AI Referral, brand mentions, conversions, GEO wins, top movers).
  • Logged in 08_reports/changelog-YYYY-MM-DD.md.
  • Monthly the 80-row AEO+GEO+AIEO retest results feed back into the report.
  • Trend lines on Looker tell the longer story; the email tells the weekly story.
Continuous loop

Once live, none of this needs re-setup. New conversions, new search clicks, new AI referrals, new mentions stream into the dashboard automatically. The client opens it anytime; the report lands every Monday.
